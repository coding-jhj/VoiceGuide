# 보이스가이드 (VoiceGuide) — 강사님 설명 스크립트

---

## 1. 왜 만드나요? (문제 정의)

시각장애인은 실내에서 혼자 이동할 때 장애물을 피하기 어렵습니다.
기존 서비스들(Google Lookout, Microsoft Seeing AI 등)은 물체를 설명해주긴 하지만,
**"왼쪽에 의자가 있으니 피하세요"** 처럼 즉각적인 행동 안내는 잘 되지 않습니다.

저희는 이 지점에 집중했습니다.
단순히 물체를 설명하는 게 아니라, **장애물을 피할 수 있는 방향 정보를 음성으로 바로 알려주는 것**입니다.

---

## 2. 뭘 만드나요? (서비스 개요)

**보이스가이드**는 카메라와 음성 명령을 조합한 주변 인지 보조 서비스입니다.

사용자가 말하거나 버튼을 누르면, 카메라로 주변을 분석해서 음성으로 안내합니다.

| 기능 | 입력 | 출력 |
|---|---|---|
| 장애물 안내 | "앞에 뭐 있어?" | "정면 가까이에 의자가 있습니다." |
| 물건 찾기 | "핸드폰 찾아줘" | "핸드폰은 오른쪽에 있습니다." |
| 물건 확인 | "이거 뭐야?" | "카메라 중앙의 물건은 가방입니다." |

**핵심 기능 우선순위**

시각장애인 입장에서 가장 중요한 기능 순서로 개발합니다.

1. **장애물 안내** — 이동 중 즉각적인 충돌 방지, 안전과 직결
2. **물건 찾기** — 일상에서 가장 자주 쓰이는 기능
3. **물건 확인** — 손에 든 물체 확인, 보조적 기능

**서비스 플랫폼**

현재는 Python + Gradio 기반 프로토타입으로 구현합니다.
실제 서비스는 **Android 앱으로 전환**할 계획이며, 공모전 제출 시 앱 설계도를 함께 포함합니다.

> 시각장애인이 실제로 Gradio 웹앱을 쓰기 어렵다는 점을 인지하고 있으며, Android 전환을 고도화 계획의 핵심으로 잡고 있습니다.

## 3. 어떻게 만드나요? (기술 구조)

### 전체 파이프라인

```
사용자 음성 입력 ("앞에 뭐 있어?")
    ↓ STT — SpeechRecognition
텍스트 변환
    ↓ 키워드 매칭 (if/elif)
모드 선택 (장애물 / 찾기 / 확인)
    ↓ YOLO11n — 객체 탐지
물체 종류 + bounding box 위치
    ↓ 방향 판단 + 거리 판단
"왼쪽 / 정면 / 오른쪽" + "가까이 / 멀리"
    ↓ 안내 문장 생성
"의자가 왼쪽 가까이에 있습니다"
    ↓ TTS — gTTS
음성 출력
```

### 음성과 이미지는 완전히 별도로 처리됩니다

YOLO는 **카메라 영상 전용**입니다. 이미지에서 물체 위치만 찾습니다.
음성 인식(STT)과 음성 출력(TTS)은 **완전히 별개 모듈**로 처리됩니다.
두 경로가 합쳐지는 지점은 최종 통합 단계(app.py)입니다.

## 4. 인식 대상 사물 (실험 범위)

10일 데모에서는 아래 **5종만** 집중합니다.

| 한국어 | YOLO 클래스명 | 선정 이유 |
|---|---|---|
| 사람 | person | 실내에서 가장 흔한 장애물 |
| 의자 | chair | 실내 이동 중 충돌 위험 1순위 |
| 테이블 | dining table | 실내 이동 중 충돌 위험 2순위 |

> 5종에 집중하는 이유: 인식률을 높이고 데모 실패 리스크를 줄이기 위해서입니다.
> 추가 확장은 데모 완성 후 공모전 고도화 단계에서 진행합니다.

## 5. 모델 학습이 필요한가요?

**필요 없습니다.**

YOLO11n은 COCO 데이터셋으로 사전학습된 모델입니다.
사람, 의자, 테이블, 가방, 휴대폰 모두 COCO 80종 안에 포함되어 있어서
**별도 학습 없이 바로 사용 가능**합니다.

```python
from ultralytics import YOLO
model = YOLO("yolo11n.pt")  # 이 한 줄로 끝
```

**파인튜닝을 하지 않는 이유:**
단순히 시간이 부족해서가 아니라, COCO 사전학습 모델의 5종 인식률이 이미 충분히 높기 때문입니다.
YOLO11n 기준 COCO 검증셋에서 person 57%, chair 47% 수준의 mAP를 보이며,
실내 환경 테스트 이미지로도 안정적으로 탐지됩니다.

STT와 TTS도 마찬가지로 별도 학습이 필요 없습니다.
Google에서 제공하는 라이브러리를 그대로 사용합니다.

- **STT**: `SpeechRecognition` 라이브러리 → 내부적으로 Google Speech API 호출, 한국어 지원
- **TTS**: `gTTS` (Google Text-to-Speech) → 텍스트를 한국어 음성으로 변환, 자연스러운 발음

```python
# STT — 음성을 텍스트로
import speech_recognition as sr
r = sr.Recognizer()
text = r.recognize_google(audio, language="ko-KR")

# TTS — 텍스트를 음성으로
from gtts import gTTS
tts = gTTS("의자가 왼쪽 가까이에 있습니다", lang="ko")
```

| 걱정 | 결론 |
|---|---|
| 모델 학습이 필요한가? | ❌ COCO 사전학습 모델 그대로 사용, 인식률 충분 |
| STT/TTS 학습이 필요한가? | ❌ Google 라이브러리 그대로 사용 |
| LLM을 써야 하나? | ❌ 키워드 매칭이 더 빠르고 안정적 |
| 데이터 수집이 필요한가? | ❌ 테스트 이미지 몇 장이면 충분 |

## 6. 기존 서비스와 뭐가 다른가요? (차별점)

| 서비스 | 주요 기능 | 한계 |
|---|---|---|
| Google Lookout | 주변 물체/텍스트 인식 | 방향 안내 없음 |
| Microsoft Seeing AI | 텍스트, 사람, 물체 설명 | 즉각적 장애물 회피 안내 없음 |
| Be My Eyes | 사람/AI 기반 시각 도움 | 사람 연결 필요, 즉시성 낮음 |

**보이스가이드 차별점 3가지**

1. **음성 명령 기반**: "핸드폰 찾아줘"처럼 말하면 모드가 자동으로 바뀝니다.
2. **방향 중심 안내**: 물체 이름만이 아니라 "왼쪽 가까이"처럼 행동 가능한 정보를 제공합니다.
3. **장애물 회피 특화**: 단순 설명이 아니라 피해야 할 위치 안내에 집중합니다.

## 7. 서버 구조

### 현재 프로토타입 구조 (서버 없음)

모든 처리가 **로컬 PC에서** 실행됩니다. STT와 TTS만 Google 서버를 잠깐 호출합니다.

```
사용자 PC
├── 카메라 입력
├── YOLO11n 탐지 (로컬 실행)
├── 방향/거리 계산 (로컬 실행)
├── STT → Google Speech API 호출 (인터넷 필요)
├── TTS → Google TTS API 호출 (인터넷 필요)
└── Gradio UI (localhost:7860, 로컬 웹서버)
```

### 예상 질문 대비

**Q. 서버는 어떻게 구성했나요?**
> "현재 프로토타입은 별도 서버 없이 로컬에서 실행됩니다. YOLO 탐지와 방향/거리 계산은 로컬 PC에서 처리하고, STT와 TTS는 Google API를 호출합니다. Gradio로 로컬 웹 서버를 띄워 UI를 제공합니다."

**Q. 실제 서비스하려면 서버가 필요하지 않나요?**
> "네, 실제 앱 서비스로 전환할 경우 YOLO 추론을 서버에서 처리하거나 모바일 온디바이스로 경량화하는 방향이 필요합니다. 현재는 프로토타입 단계라 로컬 실행으로 대체했습니다."

**Q. 인터넷이 끊기면 어떡하나요?**
> "STT와 TTS가 Google API를 사용하기 때문에 인터넷이 필요합니다. 오프라인 대안으로 TTS는 `pyttsx3`, STT는 `Whisper` 로컬 모델로 대체할 수 있습니다."

### MVP (Minimum Viable Product) 목표

> "완벽하지 않아도 되니, 핵심 기능 하나가 실제로 돌아가는 것"

```
카메라로 의자 찍기 → YOLO 탐지 → "의자가 왼쪽에 있습니다" 음성 출력
```

이것 하나가 돌아가면 MVP 완성. 나머지 기능은 그 다음 단계.

### 업무 분장 (5명)

| 역할 | 담당 업무 |
|---|---|
| **정환주** | Android 앱 UI 설계 + 카메라 입력 연결 + 추후 iOS/안경 확장 계획 설계 |
| **신유득** | YOLO 추론 서버 구축 + API 연결 + 로컬 실행 → 서버 전환 |

| **임명광** | 문제 정의 + 자료조사 + 시각장애인 영상 분석 + PPT + 발표 |
