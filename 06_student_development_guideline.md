# VoiceGuide — 학생용 개발 개선 가이드

> 대상: VoiceGuide 개발팀    
> 목적: 학생들이 다음 커밋에서 무엇을, 어떤 순서로, 어디까지 고치면 되는지 안내  
> 일자: 2026-04-29

---

## 먼저 전하고 싶은 말

VoiceGuide는 아이디어가 좋고, 사용자의 실제 불편을 해결하려는 방향도 분명합니다. "물체를 설명하는 앱"이 아니라 "어느 방향으로 피해야 하는지 말해주는 앱"으로 접근한 점은 좋은 제품 감각입니다.

다만 지금은 기능이 많아진 만큼, 실제로 동작하는 것과 문서에 적힌 것이 조금씩 어긋나기 시작했습니다. 발표 전에는 기능을 더 늘리기보다 **작지만 확실하게 되는 핵심 기능을 안정화**하는 것이 더 중요합니다.

이 문서는 Product Engineer로써 다음 작업을 어떻게 진행하면 프로젝트가 더 믿을 만해지는지 알려주는 실행 가이드입니다.

---

## 개발 진행 원칙

### 1. 기능 추가보다 검증을 먼저 합니다

지금 필요한 것은 "새 기능 5개"가 아니라 "이미 만든 핵심 기능이 실제 기기에서 안정적으로 동작한다"는 증거입니다.

발표 전 우선순위는 다음과 같습니다.

1. 실제 되는 기능 확인
2. 문서와 코드 일치
3. 앱 빌드/서버 실행 재현
4. 테스트 통과
5. 그다음 기능 개선

### 2. 안전 앱에서는 과장 표현을 줄입니다

VoiceGuide는 시각장애인 보행 보조 앱입니다. 이런 앱에서는 "대략 가능"한 기능을 "완료"라고 말하면 위험합니다.

특히 다음 기능은 표현을 조심해야 합니다.

- 거리 추정
- 신호등 판단
- 계단/낙차 감지
- SOS
- 차량 접근 경고
- 하차 알림

권장 표현:

- "정확한 거리 측정"보다 "대략적 거리 추정"
- "신호등을 판단합니다"보다 "신호등 색상을 보조적으로 안내합니다"
- "안전하게 길을 안내합니다"보다 "보행 보조 정보를 제공합니다"

### 3. MVP를 고정합니다

이번 발표/제출 기준 MVP는 다음 정도로 고정하는 것을 권장합니다.

1. 카메라로 장애물 탐지
2. 방향 안내
3. 대략적 거리 안내
4. 자연스러운 한국어 TTS 안내
5. 서버 연결 실패 시 온디바이스 fallback
6. 서버 연결 시 대시보드/GPS 시연

다음 기능은 2차 기능 또는 실험 기능으로 분리하는 편이 좋습니다. 목표 일정 D-2까지 테스트되지 않으면 발표 범위에서 제외합니다.
- 옷 매칭
- 지폐 인식
- 약 알림
- 하차 알림
- SOS
- 신호등 판단
- 버스 번호 인식
- 공간 기억

다만 안전 앱에서는 "많지만 불확실한 기능"보다 "적지만 확실한 기능"이 훨씬 강합니다.

## 권장 작업 순서

### Phase 1. 프로젝트 정리

**왜 이 Phase가 필요한가**: 외부 평가자(강사, 보호자, 다른 팀원)가 처음 저장소를 열었을 때 "어디서 시작해야 하지?"를 5분 안에 답할 수 없으면, 코드 품질과 무관하게 신뢰를 잃습니다. 진입점이 3개(`app.py`, `src/api/main.py`, `서버_DB*/main.py`)인 지금이 그런 상태입니다.

목표: 새 팀원이 봐도 어떤 서버를 실행해야 하는지 헷갈리지 않게 만든다.

해야 할 일:

1. `서버_DB/`, `서버_DB_수정/`의 역할을 결정합니다.
2. 실제 Android 앱이 호출하는 서버는 `src/api/main.py` 기준으로 단일화합니다.
3. RoadGlass/LaneWear처럼 VoiceGuide와 무관한 코드는 별도 저장소나 `legacy/`로 분리합니다.
4. README 첫 화면에 "실행 진입점" 표를 추가합니다.

원인 추정 — 왜 이렇게 됐을까:
백엔드/DB 담당이 둘로 나뉘어 각자 `main.py`를 작성했고, 머지 시점에 "어느 쪽을 본체로 둘지" 결정을 미룬 것으로 보입니다. 그러는 사이 Android 앱은 `src/api/main.py`를 호출하기 시작했고, `서버_DB*/`는 누구도 책임지지 않는 코드가 되었습니다.

교훈:
같은 책임 영역(예: API 서버)을 두 사람이 나눠 작업할 때는 **첫날에 단일 owner와 단일 진입점**을 합의해야 합니다. "나중에 머지하자"는 거의 항상 dead code를 남깁니다.

완료 기준:
- 팀원이 `README.md`만 보고 서버를 실행할 수 있습니다.
- `/detect`가 어느 파일에서 제공되는지 명확합니다.
- 중복 서버 코드 때문에 헷갈리지 않습니다.

자기 점검 질문:
- 우리 팀 외부 사람을 한 명 데려와 README만 보고 서버를 실행시켰을 때, 10분 안에 성공할까?
- 지금 dead code로 판정한 폴더를 삭제했을 때, 정말 아무도 그 코드를 쓰고 있지 않다고 자신 있게 말할 수 있는가?

### Phase 2. 문서와 실제 기능 맞추기

**왜 이 Phase가 필요한가**: 학생 발표에서 평가자가 가장 빠르게 신뢰를 잃는 순간은 "README에 됐다고 적힌 기능이 실제로는 안 될 때"입니다. 한 번 신뢰가 깨지면 잘 되는 기능까지 의심받습니다. "기능을 줄여도 좋으니 적힌 것은 무조건 된다" 상태가 평가에서 더 강합니다.

목표: "완료 기능" 목록을 실제 동작 기준으로 정리한다.

해야 할 일:

1. README의 기능 목록을 다음 세 상태로 나눕니다.
   - 동작 확인
   - 실험 기능
   - 예정 기능
2. Android assets에 실제 들어 있는 모델명을 문서와 맞춥니다.
3. `yolo11m.onnx`를 쓸 것인지, 현재 있는 `yolo11n.onnx`를 기준으로 설명할 것인지 결정합니다.
4. Depth V2 모델 파일이 없으면 "fallback 동작"이라고 명확히 적습니다.

원인 추정 — 왜 이렇게 됐을까:
기획 초기에 "이런 기능을 만들겠다"는 목록이 README에 들어갔고, 개발 중 일부 기능은 빠지거나 형태가 바뀌었지만 README는 업데이트되지 않은 것으로 보입니다. 또한 학습한 모델(`yolo11m.onnx`)과 배포 모델(`yolo11n.onnx`)이 다른 이유는 APK 용량/속도 trade-off 때문에 마지막에 경량 모델로 교체됐을 가능성이 높습니다.

교훈:
문서는 **"미래의 의도"가 아니라 "지금 검증된 사실"** 만 적어야 합니다. 의도는 `TODO.md`나 `docs/roadmap.md`에 분리하고, README는 "현재 시점에 동작이 보장되는 것"만 적는 습관을 들이세요. 이 분리가 평가자뿐 아니라 6개월 뒤의 자기 자신에게도 친절합니다.

완료 기준:

- README에 적힌 "완료" 기능은 APK나 서버에서 실제로 확인 가능합니다.
- 발표자가 문서와 다른 말을 하지 않습니다.
- 평가자가 질문했을 때 "이건 실험 기능입니다"라고 구분해 설명할 수 있습니다.

자기 점검 질문:

- README의 "완료" 항목 중 한 개를 무작위로 골라 즉석에서 시연했을 때 100% 동작하는가?
- 모델 파일명/버전이 README, `assets/`, 학습 스크립트 3곳에서 모두 일치하는가?

### Phase 3. Android 안정화

**왜 이 Phase가 필요한가**: 시연이 5분을 넘기면 발열·배터리·TTS 겹침이 누적되어 시연 자체가 멈춥니다. "데모는 됐는데 실사용은 안 된다"는 인상을 주면, 안전 보조 앱으로서의 가치는 0에 가까워집니다. 모바일 안정성은 기능이 아니라 **제품의 전제 조건**입니다.

목표: 실제 기기에서 발열, 배터리, TTS 겹침, 권한 피로를 줄이는 노력이 필요하다.

해야 할 일:

1. 캡처 주기를 현실화합니다.
2. Voting 로직을 설명과 맞춥니다.
3. 긴급 발화는 네트워크 TTS가 아니라 OS TTS로 고정합니다.
4. 권한 요청을 기능별로 나눕니다.
5. STT 미인식 fallback을 더 안전하게 바꿉니다.

원인 추정 — 왜 이렇게 됐을까:
`INTERVAL_MS = 50L`(20fps)로 설정된 이유는 개발 초기에 "탐지 결과가 부드럽게 보이는" 화면을 만들고 싶었기 때문일 가능성이 높습니다. 또한 `VOTE_MIN_COUNT = 1`은 빠른 디버깅 중에 "왜 안내가 안 나오지?" 문제를 풀려고 임시로 낮췄다가 원복하지 않은 흔적으로 보입니다. 권한 일괄 요청은 "한 번에 다 받자"가 코드는 짧지만, 사용자 입장에서는 가장 거부감이 큰 패턴입니다.

교훈:
**개발 편의로 낮춘 임계값은 반드시 기록**해 두고, 시연 전 체크리스트에 "임시값 원복" 항목을 넣으세요. 그리고 모바일에서 "화면이 부드럽다"는 결정은 거의 항상 발열·배터리와 trade-off입니다 — 시각장애인 사용자에게 화면 부드러움은 0의 가치입니다. 사용자가 누구인지에 따라 trade-off의 방향이 달라진다는 감각이 핵심입니다.

완료 기준:

- 앱을 5분 이상 실행해도 과도한 발열이 없습니다.
- 같은 문장이 계속 겹쳐 나오지 않습니다.
- SMS 권한은 SOS 기능을 실제로 설정할 때만 요청합니다.
- 모르는 음성 명령 때문에 갑자기 장애물 안내가 시작되지 않습니다.

자기 점검 질문:

- 실제 기기에서 5분 연속 시연했을 때 기기 뒷면 온도가 시작 시 대비 얼마나 올랐는가? (40도 이상이면 사용자가 손에 들기 어렵습니다)
- 앱을 처음 실행한 사용자가 권한 4개를 한 번에 요청받으면 "이 앱이 왜 SMS를 필요로 하지?"라고 물을 수 있는가? 그 질문이 기능 흐름 안에서 자연스럽게 답변되는가?

### Phase 3.5. 음성 안내 UX 개선

**왜 이 Phase가 필요한가**: 시각장애인 사용자에게 음성은 화면을 가진 사용자의 **UI 그 자체**입니다. UI가 시끄럽고, 반복되고, 우선순위가 없으면 사용자는 앱을 끕니다. 그리고 끄지 않더라도 "위험"이라는 단어가 자주 나오면 진짜 위험할 때 의미가 사라집니다 — 늑대와 양치기 소년 효과입니다.

목표: "탐지 결과를 읽어주는 앱"이 아니라 "사람이 바로 이해할 수 있게 말해주는 앱"으로 만든다.

영상 확인 결과, 현재 앱은 탐지와 TTS가 실제로 동작한다는 점은 좋다. 하지만 음성 안내가 자연스럽지 않고, 같은 표현이 반복되거나, 위험하지 않은 물체에도 "위험", "멈추세요", "피해가세요"가 붙어 사용자가 불안해질 수 있다.

해야 할 일:

1. 음성 문장을 더 짧게 만듭니다.
2. 생활 물체와 위험 물체의 말투를 분리합니다.
3. 같은 문장이나 같은 거리 표현을 반복하지 않습니다.
4. "위험", "멈추세요", "피해가세요"는 정말 위험한 상황에만 사용합니다.
5. 찾기 모드와 장애물 모드의 발화 정책을 분리합니다.

원인 추정 — 왜 이렇게 됐을까:
탐지된 모든 물체에 동일한 "방향 + 거리 + 행동 지시" 템플릿이 적용된 것으로 보입니다. 개발 초기에는 "탐지 결과가 음성으로 나온다"는 자체가 성취였고, 클래스별 톤 분기는 "나중에 다듬자"로 미뤄진 결과입니다. 또한 NLG에서 "정보를 빠뜨리지 말자"는 본능적 욕구가 반대 방향(짧고 우선순위 있게)을 막고 있을 가능성이 높습니다.

교훈:
**음성 UI 설계의 핵심은 "무엇을 말할까"가 아니라 "무엇을 말하지 않을까"입니다.** 시각 UI는 사용자가 무시할 자유가 있지만, 음성은 강제로 들립니다. 그래서 음성 정보의 비용은 시각 정보보다 훨씬 비쌉니다. 또한 안전 관련 단어("위험", "멈추세요")는 **희소성이 의미를 만듭니다** — 자주 쓰면 단어 자체의 신호 가치가 0으로 수렴합니다.

완료 기준:

- 책상 위 `키보드`, `마우스`, `TV`, `책`에는 기본적으로 긴급 표현을 쓰지 않습니다.
- "더 가까이", "매우 가까이" 같은 표현이 연속 반복되지 않습니다.
- 한 번의 안내가 1~2문장 안에서 끝납니다.
- 사용자가 듣기에 기계가 탐지 결과를 나열하는 느낌보다, 사람이 짧게 알려주는 느낌에 가깝습니다.

권장 문장 예시:

| 상황 | 현재처럼 들릴 수 있는 문장 | 권장 문장 |
|---|---|---|
| 책상 위 키보드 | 왼쪽 앞에 키보드가 있어요. 매우 가까이. 오른쪽으로 피해가세요. | 왼쪽 가까이에 키보드가 있어요. |
| 책상 위 모니터/TV | 위험! 바로 앞에 TV가 있어요. 멈추세요. | 정면 가까이에 모니터가 있어요. |
| 찾기 모드 | 왼쪽에 키보드도 있어요. 오른쪽 앞에 마우스도 있어요. | 키보드는 왼쪽 가까이에 있어요. |
| 실제 위험 상황 | 정면 가까이에 자동차가 있어요. | 위험, 정면 차량. 멈추세요. |

자기 점검 질문:

- 우리 앱을 1분 이상 들었을 때 "위험"이라는 단어가 몇 번 나오는가? 그중 진짜 위험 상황은 몇 번이었는가?
- 시각장애인 친구나 사용자에게 30초 시연 영상을 들려줬을 때, "이 안내를 종일 듣고 다닐 수 있겠다"는 반응이 나오는가?

### Phase 4. ML/Vision 검증

**왜 이 Phase가 필요한가**: 안전 보조 앱에서 오탐(False Positive)은 "조금 불편한 알람"이 아니라 **사용자가 진짜 위험을 무시하게 만드는 학습 신호**입니다. 책상 위 의자를 계속 "위험"으로 안내하면 사용자는 며칠 안에 그 안내를 끄거나 무시하게 되고, 그 다음 진짜 차량 경고가 나와도 같은 반응을 합니다. mAP/Recall만 좋아 보이는 모델이 현실에서 위험한 이유입니다.

목표: 탐지 모델의 "좋아 보이는 수치"보다 실제 사용 환경에서의 오탐/누락을 확인한다.

해야 할 일:

1. Precision, Recall, F1, False Positive Rate를 함께 봅니다.
2. 계단 pseudo-label 고정 bbox 문제를 정리합니다.
3. Depth distance scale을 보정하거나, 보정 전에는 미터 표현을 조심합니다.
4. 버스 OCR은 실제 버스 사진 테스트셋을 만듭니다.

원인 추정 — 왜 이렇게 됐을까:
계단 pseudo-label에 고정 bbox `(cx=0.5, cy=0.7, w=0.9, h=0.6)`을 강제 삽입한 것은, 라벨링 시간을 줄이기 위해 "계단은 보통 화면 아래쪽에 크게 보인다"는 직관을 코드로 변환한 결과로 보입니다. 단기적으로는 mAP 수치가 빠르게 올라가지만, 모델은 "계단의 시각적 특징"이 아니라 "화면 아래쪽 큰 영역"을 학습하게 됩니다. 그래서 의자·배낭처럼 화면 아래에 큰 물체가 오면 stairs로 오탐합니다.

또한 `DEPTH_SCALE = 1.0`은 "일단 통합부터 하고 나중에 보정하자"가 미뤄진 결과로 보입니다. 평가 지표가 Recall 중심이 된 것은 "탐지 성공률"이 발표에서 직관적이기 때문이지만, 안전 앱의 진짜 적은 "탐지 실패"가 아니라 "잘못된 자신감"입니다.

교훈:
**Pseudo-label은 항상 "모델이 학습할 수도 있는 잘못된 패턴"을 함께 만듭니다.** 시간을 절약하는 모든 라벨링 트릭에는 비용이 따르고, 그 비용은 보통 오탐·편향 형태로 늦게 나타납니다. 그리고 ML 평가에서 "어떤 메트릭을 보지 않기로 결정했는가"는 "어떤 메트릭을 보기로 했는가"만큼 중요합니다 — 안 보는 메트릭에 모델은 자유롭게 망가집니다.

완료 기준:

- "어떤 환경에서 잘 되고, 어떤 환경에서 약한지"를 말할 수 있습니다.
- 계단 mAP 같은 수치를 과장하지 않습니다.
- 발표 자료에 실제 테스트 이미지 수와 결과를 함께 제시합니다.

자기 점검 질문:

- 우리 모델이 가장 자주 틀리는 이미지 5장을 즉시 보여줄 수 있는가? 그 5장에서 공통된 패턴이 보이는가?
- "거리 X미터"라는 안내가 나올 때, 우리는 그 수치가 실제로 ±몇 미터 오차일지 답할 수 있는가?

### Phase 5. 백엔드/DB/배포 안정화

**왜 이 Phase가 필요한가**: 이 서버는 시각장애인 사용자의 **실시간 GPS 좌표와 이동 경로**를 저장합니다. 인증 없는 공개 엔드포인트로 이게 노출되면, 단순한 보안 사고가 아니라 **사용자 신변의 위험**으로 이어집니다. 일반 웹앱과 비교해 같은 보안 결함도 결과의 무게가 다릅니다.

목표: 서버가 공개 인터넷에 올라갔을 때 위치 정보와 이미지 업로드가 위험해지지 않게 한다.

해야 할 일:

1. 공개 API에 최소한의 API Key를 추가합니다.
2. CORS를 `*`로 열지 않습니다.
3. DB credential을 코드에 하드코딩하지 않습니다.
4. `snapshots` 테이블이 무한히 쌓이지 않게 정리 정책을 넣습니다.
5. `/health`에서 DB 상태도 확인합니다.

원인 추정 — 왜 이렇게 됐을까:
DB 비밀번호 하드코딩은 "로컬에서 빠르게 돌리기 위해" 잠시 박아뒀다가 그대로 커밋된 흔적입니다. CORS `*`는 안드로이드 클라이언트와의 통신이 안 되는 문제를 빠르게 풀려고 "일단 다 열자"로 해결한 결과로 보입니다. 인증 부재는 학습 환경에서 흔한 패턴인데 — 부트캠프 예제 코드에 인증 미들웨어가 거의 없기 때문입니다.

교훈:
**공개 저장소에 한 번 들어간 비밀번호는 코드에서 지워도 사라지지 않습니다.** Git history에 영구 보존되므로, 발견 즉시 비밀번호를 교체하고 history rewrite(BFG/`git filter-repo`)를 동시에 해야 합니다. 그리고 "빠르게 풀려고 다 열자"는 거의 모든 보안 사고의 첫 단추입니다 — 막힌 이유를 이해하지 않고 우회하면, 막힌 이유가 보안 설계였을 때 그 설계가 무너집니다. 다음부터는 "왜 막혀 있는가?"를 먼저 물어보는 습관을 들이세요.

완료 기준:

- 서버 URL을 알아도 아무나 GPS 상태를 조회할 수 없습니다.
- `.env` 없이 소스코드에 DB 비밀번호가 남아 있지 않습니다.
- 장시간 실행해도 DB가 불필요하게 커지지 않습니다.

자기 점검 질문:

- 지금 저장소를 GitHub에서 공개 검색했을 때, 검색 결과로 비밀번호·API 키가 한 줄도 나오지 않는가? (이미 노출됐다면 history rewrite + 키 교체를 즉시 했는가?)
- 우리 서버 URL을 모르는 친구에게 알려줬을 때, 그 친구가 시각장애인 사용자의 위치를 조회할 수 있는가? 없다면 무엇이 막아주고 있는가?

### Phase 6. 테스트와 실행환경 고정

**왜 이 Phase가 필요한가**: "내 컴퓨터에서는 됐는데"는 팀 프로젝트에서 가장 비싼 디버깅 비용을 만듭니다. 발표 당일에 강사 노트북에서 빌드가 안 되거나, 다른 팀원 환경에서 테스트가 깨지면 그동안의 작업이 평가에 반영되지 않습니다. 재현 가능성은 코드 품질이 아니라 **존재 증명**의 문제입니다.

목표: 강사/팀원/배포 환경에서 같은 명령으로 같은 결과가 나오게 한다.

해야 할 일:

1. Python 3.10 기준 venv 생성 방법을 고정합니다.
2. `pytest` 기본 테스트가 서버 없이도 돌아가게 합니다.
3. live server 테스트는 `integration` 마커로 분리합니다.
4. Android용 `gradlew`를 추가해 macOS/Linux에서도 빌드 가능하게 합니다.
5. `test_imports.py`에서 Gradio/pygame 같은 데모 의존성을 핵심 테스트에서 분리합니다.

원인 추정 — 왜 이렇게 됐을까:
`gradlew`(Unix용 실행 파일)가 빠지고 `gradlew.bat`(Windows용)만 있는 것은 팀원 한 명이 Windows 환경에서 작업하고 그 결과를 그대로 커밋한 흔적입니다. `test_imports.py`에 Gradio/pygame이 섞인 것은 "import만 되면 일단 테스트 통과"로 빠르게 만든 뒤 분리하지 않은 결과입니다. live server 테스트와 unit 테스트가 섞인 것은 처음에 "테스트가 있다"는 사실이 중요했고, 종류 구분은 나중 문제로 미뤄진 흔적입니다.

교훈:
**테스트는 "내 환경에서 통과"가 아니라 "남의 환경에서 통과"가 가치입니다.** 그리고 테스트는 종류별로 비용이 다릅니다 — unit 테스트는 1초, integration 테스트는 10초, e2e 테스트는 1분. 마커로 분리하지 않으면 가장 느린 테스트의 비용을 매번 모두가 치르게 되고, 결국 아무도 테스트를 안 돌리게 됩니다. CI/CD를 도입하기 전에 **로컬에서 "빠른 것 / 느린 것" 분리**부터 하세요.

완료 기준:

```bash
python -m pytest
```

가 기본 환경에서 통과합니다.

```bash
cd android
./gradlew :app:assembleDebug
```

가 macOS/Linux에서도 실행됩니다.

### 6. `src/nlg/sentence.py`

문제:
NLG는 잘 만들어졌지만, 안전 상황에서는 더 보수적으로 짧게 말해야 합니다.

영상에서 확인한 추가 문제:

- 음성이 자연스럽지 않고, 탐지된 물체를 그대로 이어 붙인 느낌이 납니다.
- `키보드`, `마우스`, `TV`, `책` 같은 생활 물체에도 "멈추세요", "피해가세요"가 붙어 과하게 들립니다.
- "매우 가까이", "가까이", "더 가까이" 같은 거리 표현이 반복되어 사용자가 핵심 정보를 놓칠 수 있습니다.
- 카메라가 다른 곳을 보고 있는데 이전 안내가 남아 있는 듯한 구간이 있어 신뢰감을 떨어뜨립니다.

수정 방향:

- 차량 critical 임계값을 8m에서 12m 이상으로 검토합니다.
- 계단/낙차 hazard는 다른 물체와 합치지 말고 단독 발화합니다.
- 차량/계단/칼/가위 등은 항상 짧은 문장으로 우선 발화합니다.
- 생활 물체(`키보드`, `마우스`, `TV`, `책`, `컵` 등)는 기본적으로 정보성 안내만 합니다.
- 위험 표현은 위험 클래스와 바닥 장애물에만 사용합니다.
- 같은 거리 표현을 여러 번 붙이지 말고, 가장 중요한 물체 1개만 우선 말합니다.

권장 정책:

| 물체/상황 | 안내 레벨 | 문장 톤 |
|---|---|---|
| 차량, 오토바이, 자전거 접근 | 긴급 | "위험, 정면 차량. 멈추세요." |
| 계단, 낙차, 턱 | 긴급 | "계단이 있어요. 멈추세요." |
| 칼, 가위, 유리 파편 | 긴급/주의 | "오른쪽 가까이에 칼이 있어요. 조심하세요." |
| 바닥의 가방, 박스, 의자 다리 | 주의 | "왼쪽 바닥에 가방이 있어요." |
| 키보드, 마우스, TV, 책 | 정보 | "왼쪽 가까이에 키보드가 있어요." |
| 찾기 모드의 대상 물체 | 정보 | "키보드는 왼쪽 가까이에 있어요." |

완료 기준:

- 긴급 상황에서 TTS가 짧고 빠릅니다.
- "그리고 의자도 있어요" 같은 부가 정보가 위험 안내를 늦추지 않습니다.
- 책상 위 일반 물체에는 "위험", "멈추세요", "피해가세요"가 나오지 않습니다.
- 시연 영상을 다시 찍었을 때 음성 안내가 자연스럽고 덜 반복적으로 들립니다.

### 6-1. `src/voice/tts.py` / Android TTS 설정

문제:
영상 기준으로 음성이 기능적으로는 나오지만 자연스럽게 들리지는 않습니다. 말의 속도, 끊어 읽기, 반복 억제, 문장 길이가 함께 조정되어야 합니다.

수정 방향:

- 긴급 안내는 OS TTS로 빠르게 재생합니다.
- 일반 안내는 너무 빠르지 않게, 한 문장을 짧게 유지합니다.
- 문장 끝마다 짧은 쉼을 두되, 여러 문장이 길게 누적되지 않게 합니다.
- 같은 문장을 3~5초 안에 반복하지 않습니다.
- TTS 음색이나 속도는 설정값으로 분리하는 것을 권장합니다.

완료 기준:

- 사용자가 한 번 듣고 바로 행동할 수 있습니다.
- 안내가 기계적으로 나열되지 않습니다.
- 반복 발화 때문에 피로감이 생기지 않습니다.

권장 발화 원칙:

1. 긴급은 짧게: "위험, 정면 차량. 멈추세요."
2. 정보는 부드럽게: "왼쪽 가까이에 키보드가 있어요."
3. 찾기는 대상만: "키보드는 왼쪽 가까이에 있어요."
4. 중복은 억제: 같은 물체를 계속 말하지 않기
5. 확신 없으면 낮은 톤: "정면에 물체가 있는 것 같아요."

## Phase 7. 프로덕트 디자인 / 프런트엔드 UI

**왜 이 Phase가 필요한가**: 시연 영상에서 가장 빠르게 평가자의 첫인상을 결정하는 것은 코드 품질도, ML 정확도도 아니라 **앱 첫 화면의 시각적 정돈도**입니다. 현재 앱은 기능적으로는 동작하지만, 화면이 "데모 프로토타입" 단계에 머물러 있어 결과물이 실제보다 미완성으로 보입니다. 1-2시간 분량의 디자인 정리만으로 발표 인상이 크게 바뀝니다.

또한 시각장애인 보조 앱이라고 해서 시각 디자인이 불필요한 것이 아닙니다. 화면을 보는 사람은 ① **보호자 / 가족** ② **현장에서 도와주는 비서** ③ **저시력 사용자(LV)** ④ **앱을 평가하는 강사·심사위원**입니다. 이들 각자에게 화면이 의미 있게 정돈되어야 합니다.

목표: "데모 프로토타입"처럼 보이는 현재 화면을 "출시 가능한 안전 보조 앱"으로 인상이 바뀌도록 정돈한다.

### 영상 기반 디자인 평가 — 관찰된 화면 요소

영상에서 확인한 주요 UI 구성은 다음과 같습니다.

| 영역 | 현재 상태 | 평가 |
|---|---|---|
| 워드마크 | 화면 상단 좌측 "VoiceGuide", 핸드라이팅 계열 폰트 | 🟡 가독성 낮음. 시각 보조 앱에서 글자 가독성은 신뢰도 신호 |
| 디버그 URL | 워드마크 아래 `http://172.30.1.70:8000` 그대로 노출 | 🔴 시연·발표·실사용 모두에서 부적절 |
| 카메라 프리뷰 | 화면 중앙, 약 60% 차지 | 🟢 면적 배분 적절 |
| bbox 오버레이 | 빨강/녹색/시안/노랑이 클래스/프레임마다 다르게 배정 | 🔴 색이 위험도를 표현하지 못함 |
| confidence 라벨 | "TV 78%", "마우스 85%" 형태로 노출 | 🟡 사용자에게 의미 없는 디버그 정보 |
| 라벨 언어 | 한국어("마우스") + 영문 클래스명("TV") 혼재 | 🟡 일관성 부족 |
| 안내 문구 | 하단 검정 배경 + 흰 글자, 가운데 정렬 | 🟢 콘트라스트 좋음. 단 4줄까지 길어지는 경우 있음 |
| 모드 표시 | "모드: 찾기 \| 방향: 정면" 한 줄 메타 | 🟢 정보 밀도 적절 |
| 상태 ("분석 중…") | 위치가 프레임마다 달라짐 (프리뷰 안 / 텍스트 위) | 🟡 일관된 자리 필요 |
| 하단 버튼 | "분석 중지"(파랑) / "음성 명령"(녹색) 동일 크기 | 🟡 위험 액션과 주 액션이 시각적으로 동등 |
| 시스템 상태바 | 시계·Wi-Fi·배터리 그대로 노출 | 🟢 일반 앱 기준 정상. 발표 영상에선 가리는 게 깔끔 |

### 원인 추정 — 왜 이렇게 됐을까

이 프로젝트에 명시적 "디자인 담당"이 지정되지 않은 것으로 보입니다. Android 개발자가 기능을 빠르게 붙이면서 디버그용으로 띄운 텍스트(IP 주소, confidence %, 워드마크 prototype 폰트)가 그대로 남았고, bbox 색은 OpenCV 기본 색 또는 클래스 인덱스에 따라 자동 배정된 것으로 보입니다. 안내 문구의 길이와 폰트는 "일단 보이게" 단계에서 멈췄습니다.

또한 시각장애인 보조 앱이라는 정체성이 "그러면 화면은 안 봐도 된다"는 잘못된 결론으로 이어진 흔적이 있습니다 — 실제로는 화면을 보는 다양한 사용자(보호자·시연자·심사위원·저시력 사용자)가 존재합니다.

### 교훈

**디자인 정돈은 기능 추가가 아니라 "기능 빼기"입니다.** 화면에서 의미 없는 정보(IP, 콘피던스 %, 핸드라이팅 폰트)를 제거하는 것만으로도 같은 코드의 인상이 완전히 달라집니다. 또한 색은 안전 앱에서 **언어**입니다 — 무작위 색은 "정보 없음"이지만, 일관된 색 매핑은 사용자(특히 저시력)에게 음성 안내 도착 전에 위험도를 미리 알리는 보조 채널이 됩니다.

### 해야 할 일 (우선순위 순)

#### 1. 디버그 정보 제거 (예상 30분)

- 화면 상단 `http://172.30.1.70:8000` URL 제거 또는 설정 화면으로 이동
- bbox 라벨에서 `XX%` 부분 숨김. 디버그 모드 토글(`BuildConfig.DEBUG`)에서만 표시
- 워드마크 폰트를 **Pretendard** 또는 **Noto Sans KR Bold**로 교체

완료 기준: 첫 화면 진입 시 IP·% 등 개발자만 의미 있는 문자열이 보이지 않음.

#### 2. bbox 색상을 위험도 언어로 통일 (예상 1시간)

| 색상 | 의미 | 적용 클래스 |
|---|---|---|
| 빨강 (#E53935) | 긴급 (critical) | 차량, 오토바이, 자전거, 계단, 낙차 |
| 노랑 (#FDD835) | 주의 (caution) | 칼·가위·유리 등 위험 도구, 가까운 바닥 장애물 |
| 녹색 (#43A047) | 정보 (info) | 키보드·마우스·TV·책 등 생활 물체 |
| 흰색 (#FFFFFF) | 찾기 모드 대상 | 사용자가 음성으로 지정한 물체 |

이 매핑은 [src/nlg/sentence.py]의 `get_alert_mode` 결과와 동일한 분류를 써야 합니다 — 음성과 시각이 같은 위험도 신호를 보내야 합니다.

완료 기준: 같은 클래스가 다른 프레임에서 다른 색으로 표시되지 않음. 빨간 박스가 보이면 음성도 긴급 톤.

#### 3. 안내 문구 영역 압축 (예상 1시간)

- 하단 텍스트 영역을 **최대 2줄**로 제한
- 긴 안내는 첫 문장만 노출하고 "더보기" 토글 또는 자동 페이드아웃
- 폰트 크기는 24sp 이상, line-height 1.4 이상 유지 (저시력 사용자 고려)
- 폰트는 **Pretendard SemiBold** 또는 **Noto Sans KR Medium**

완료 기준: 어떤 안내가 나와도 카메라 프리뷰 영역이 잘리지 않음.

#### 4. 하단 버튼 위계 분리 (예상 30분)

| 버튼 | 현재 | 개선 |
|---|---|---|
| 음성 명령 (주 액션) | 녹색, 동일 크기 | 큰 primary 버튼, 색상 강조, 좌측 또는 화면 폭 60% |
| 분석 중지 (위험 액션) | 파랑, 동일 크기 | 작은 outline 버튼 또는 회색, 우측 |

이유: "음성 명령"이 사용자가 가장 자주 쓰는 액션이므로 주 액션이고, "분석 중지"는 의도된 종료 행위 — 두 개가 시각적으로 동등하면 사용자가 잘못 누를 가능성이 높아집니다. 특히 시각장애인 사용자가 화면을 만질 때 큰 영역이 음성 명령이어야 합니다.

완료 기준: 화면을 보지 않고도 "큰 쪽이 음성 명령"이라는 지문 감각이 만들어짐.

#### 5. 접근성(a11y) 보강 (예상 2시간)

- 모든 버튼에 `contentDescription` 추가 (TalkBack 대응)
- 터치 영역 최소 60dp × 60dp 보장
- 주 액션 버튼 long-press 시 진동 피드백 추가 (저시력 사용자가 위치 학습 가능)
- 다크 모드 전제 + 시스템 폰트 크기 200%까지 깨지지 않도록 ConstraintLayout 사용

완료 기준: TalkBack을 켜고 화면을 손가락으로 훑었을 때 모든 핵심 요소가 음성으로 읽힘.

#### 6. 워드마크/스플래시/앱 아이콘 정리 (예상 1시간)

- 워드마크: 산세리프 + 작은 마이크/지팡이 아이콘 (48dp 이내)
- 앱 아이콘: 검정 배경 + 흰 마이크 또는 흰 지팡이 (Material adaptive icon)
- 스플래시: 검정 배경 + 워드마크만 0.8초

완료 기준: 안드로이드 홈 화면에 설치된 앱 아이콘이 "프로토타입 기본 아이콘"이 아님.

### 시연 영상 촬영 가이드 (발표용)

영상 자체도 평가 대상입니다. 다음을 권장합니다.

- 시스템 상태바(시계·Wi-Fi·배터리) **숨김** — Android 시스템 설정에서 "Demo mode" 활용 가능
- 화면 녹화 시 **세로 풀화면 + 30fps 이상**으로 재녹화 (현재 영상은 90000/1 fps 메타데이터로 비정상 표기 — Samsung 화면 녹화 기본값 변경 필요)
- 시연 시나리오 3개 미리 정해서 같은 흐름으로 촬영:
  1. **장애물 모드**: 책상 위 키보드/마우스 → 정보성 안내
  2. **찾기 모드**: "키보드 찾아줘" → 위치 안내
  3. **위험 시뮬레이션**: 의자 다리·계단 사진 출력물로 긴급 톤 시연
- 영상 시작 3초는 **검정 + 워드마크 + "VoiceGuide — 시각장애인 보행 보조 앱"** 자막

### 자기 점검 질문

- 우리 앱 첫 화면을 5초 동안 보여줬을 때, 보호자가 "이게 뭐 하는 앱이지?"가 아니라 "안전 보조 앱이구나"를 알아챌 수 있는가?
- 화면을 보지 않고 손가락만으로 화면 하단을 만졌을 때, "음성 명령"과 "분석 중지"를 구분할 수 있는가?
- 빨간 bbox가 보이는 순간 음성도 긴급 톤이 나오는가? 또는 둘 중 하나만 긴급이라면 그 불일치가 사용자에게 어떻게 들리는가?

## 팀 역할과 실행 분담

이 프로젝트는 팀장이 주도하는 5인 팀 과제입니다. 위에서 제안한 작업들을 발표일까지 끝내려면 **누가 무엇을 책임지는지**가 명확해야 합니다. 명확하지 않으면 모두가 모든 것을 조금씩 하다가 아무 것도 끝나지 않는 상태로 마감을 맞이합니다.

### 5인 역할 정의 (권장)

| 역할 | 주 책임 영역 | 코드 영역 | 이번 가이드 담당 Phase |
|---|---|---|---|
| **팀장 (Tech Lead / PM)** | 일정·통합·의사결정·발표 흐름 | `README.md`, `docs/`, 통합 PR 머지 | Phase 1, Phase 2, 발표 체크리스트 |
| **Android / UX 담당** | 모바일 앱 안정화, UI 정돈 | `android/app/`, `MainActivity.kt`, `YoloDetector.kt` | Phase 3, Phase 7 |
| **Vision / ML 담당** | 탐지·Depth·OCR·평가 | `src/depth/`, `src/vision/`, `src/ocr/`, `train/`, `tools/benchmark.py` | Phase 4 |
| **Backend / DB 담당** | 서버, DB, 보안, 배포 | `src/api/`, `requirements.txt`, `railway.toml` | Phase 5 |
| **Voice / NLG 담당** | STT·TTS·문장 생성·시연 영상 | `src/voice/`, `src/nlg/`, 시연 시나리오 | Phase 3.5, Phase 6 일부 |

> **참고**: 5인 팀이지만 책임 영역은 6개 이상이므로, 일부 역할은 겸직합니다(예: 팀장 + Android, Voice + 시연 영상 등). 핵심은 **각 Phase에 명확한 단일 owner**가 있는 것이지, 한 사람이 한 영역만 한다는 의미가 아닙니다.

### 팀장이 직접 해야 할 것 vs 위임할 것

#### 팀장이 반드시 직접 해야 할 것

1. **Phase 1 dead code 정리 의사결정** — `서버_DB/`, `서버_DB_수정/`를 누가 만들었든 **삭제 결정은 팀장이** 합니다. 만든 사람이 미련 때문에 못 지웁니다.
2. **MVP 고정** — 어떤 기능을 발표 범위에 넣고 어떤 기능을 뺄지, 발표 D-3에 결정. **팀장이 거부권을 행사**해야 다른 팀원이 "이거 하나만 더 넣자"를 멈춥니다.
3. **README 최종 검수** — 각자 작성한 부분을 팀장이 마지막에 톤·과장 표현 검수.
4. **발표 흐름 + Q&A 대비 시나리오** — 누가 어떤 질문을 받으면 누가 답하는지 매핑.
5. **매일 5-10분 스탠드업 진행** — 어제 한 일, 오늘 할 일, 막힌 점 — 화상이든 채팅이든.

#### 팀장이 위임해야 할 것

1. **Phase별 작업 자체** — 팀장이 직접 코드를 다 짜면 통합·검수가 안 됩니다.
2. **각자 영역 내 기술 결정** — 예: bbox 색상 hex 코드, TTS 발화 속도. 팀장이 일일이 결정하면 의사결정 병목이 됩니다.
3. **테스트 작성** — 각 영역 owner가 책임. 팀장은 "테스트가 있는가"만 체크.

### 협업 원칙 (팀장 운영 가이드)

1. **단일 진입점 원칙**: 같은 영역(예: API 서버)을 두 사람이 만지지 않습니다. 첫날 owner를 정하고 PR은 owner에게 리뷰 요청.
2. **PR 사이즈 제한**: 한 PR은 200줄 이내. 더 크면 팀장이 분할 요청.
3. **의사결정 escalation**: 30분 이상 두 사람이 결정 못 내리면 팀장에게 가져옴 — 팀장이 "지금은 이렇게 가자"를 결정하고 회고에서 다시 본다.
4. **매일 commit**: 진행이 0이어도 "오늘 막힌 점" 한 줄을 commit message나 채팅에 남김. 침묵이 가장 비싼 비용.
5. **금지 행동**: 발표 D-2 "기능 추가" 금지. D-1 이후 "리팩토링" 금지.

### 팀장이 매일 던질 3개 질문

스탠드업에서 각 팀원에게 3개 질문만 던지세요.

1. **어제 끝낸 것은 무엇인가?** (말로 시연 가능한 것)
2. **오늘 끝낼 것은 무엇인가?** (구체적 PR/파일/함수)
3. **무엇이 막혀 있는가?** (팀장이 풀 수 있는 것 / 다른 팀원 도움이 필요한 것)

이 3개 질문이 모든 다른 회의를 대체합니다.

## 최종 목표

이번 프로젝트의 목표는 기능을 많이 보이는 것이 아니라, 다음 질문에 자신 있게 답하는 것입니다.

1. 실제로 어떤 기능이 안정적으로 동작하나요?
2. 동작하지 않거나 불확실한 기능은 무엇인가요?
3. 사용자의 안전을 위해 어떤 보수적 선택을 했나요?
4. 서버가 실패해도 앱은 어떻게 대응하나요?
5. 모델이 틀렸을 때 사용자가 위험해지지 않도록 어떤 장치를 두었나요?

이 다섯 질문에 답할 수 있으면 프로젝트의 신뢰도가 크게 올라갑니다.
