# 보이스가이드 자료조사 + 검증 10일 1등용 계획

## 1. 이 문서의 목적

이 문서는 보이스가이드 프로젝트의 **자료조사와 검증 파트**를 10일 안에 현실적으로 수행하면서도, 발표에서 1등을 노릴 수 있을 만큼 탄탄하게 보이도록 정리한 실행 계획이다.

기존 자료조사 문서의 장점:

- 근거 자료가 많다.
- 기존 서비스 비교가 풍부하다.
- 설문/인터뷰/테스트까지 다양한 검증 방법이 있다.

하지만 10일, 온라인 5명 팀에서는 모든 것을 다 하기 어렵다.

따라서 이 문서는 다음 원칙으로 다시 정리한다.

> 자료조사는 깊이 있게 하되, 발표에는 핵심 근거만 사용한다. 검증은 설문 대신 문헌 검증, 기존 서비스 비교, 기술 타당성 검토, 팀 내부 기능 테스트로 현실적으로 수행한다.

---

## 2. 최종 목표

자료조사와 검증 파트의 최종 목표는 다음 4가지를 증명하는 것이다.

1. **문제의 실제성**
   - 시각장애인/저시력 사용자는 주변 인지와 실내 이동에서 실제 어려움을 겪을 수 있다.

2. **서비스 필요성**
   - 기존 방식과 기존 서비스만으로는 모든 상황을 해결하기 어렵다.

3. **기술 적합성**
   - STT, TTS, 카메라 객체 탐지는 이 문제를 해결하는 데 적합한 기술 흐름이다.

4. **프로토타입 검증**
   - 대표 시나리오에서 입력 → 인식 → 판단 → 음성 안내 흐름이 작동한다.

발표용 핵심 문장:

> 시간 제약상 실제 사용자 모집 대신, 문헌 조사와 기존 서비스 비교를 통해 문제의 필요성을 확인하고, 기술 구조 검토와 팀 내부 시야 제한 테스트로 프로토타입의 기능 흐름을 검증했습니다.

---

## 3. 검증 프레임

자료조사와 검증은 다음 흐름으로 이어져야 한다.

```text
문헌 자료 조사
  ↓
사용자 문제 확인
  ↓
기존 서비스 비교
  ↓
우리 서비스 차별점 도출
  ↓
기술 타당성 확인
  ↓
프로토타입 테스트
  ↓
실패 사례와 개선 방향 정리
  ↓
PPT 반영
```

이 흐름이 발표에서 보이면 “그냥 만든 앱”이 아니라 “조사와 검증을 거친 서비스”로 보인다.

## 4. 자료조사 핵심 질문

자료조사는 무작정 많이 찾는 것이 아니라, 아래 질문에 답하기 위해 한다.

### 질문 1. 누가 문제를 겪는가?

답:

- 시각장애인
- 저시력 사용자
- 화면 중심 앱 사용이 어려운 사용자

### 질문 2. 무엇이 불편한가?

답:

- 낯선 실내 공간 이동
- 주변 장애물 확인
- 물건 위치 파악
- 손에 든 물건 확인
- 화면 조작 중심 앱 사용

### 질문 3. 왜 이 문제가 중요한가?

답:

- 이동 안전과 관련된다.
- 독립적인 생활과 관련된다.
- 주변 사람에게 계속 도움을 요청해야 하는 부담과 관련된다.

### 질문 4. 기존 서비스로 충분한가?

답:

- 기존 서비스는 시각 정보 설명, 텍스트 읽기, 사람 연결, AI 이미지 설명에 강점이 있다.
- 하지만 우리 프로젝트는 사용자의 음성 명령에 따라 장애물 안내, 물건 찾기, 물건 확인을 하나의 흐름으로 연결하는 점을 강조한다.

### 질문 5. 우리 기술 선택이 적합한가?

답:

- STT는 화면 조작 없이 명령을 입력하게 한다.
- TTS는 시각 정보를 음성으로 바꿔 전달한다.
- 카메라 객체 탐지는 주변 물체를 인식한다.
- 방향 판단은 사용자가 행동할 수 있는 정보로 바꿔준다.

## 5. 사용할 핵심 근거 자료

PPT에는 너무 많은 출처를 넣지 않는다.

가장 설득력 있는 근거 5개만 사용한다.

### 5-1. WHO 시각장애 및 시력 손상 자료

핵심 내용:

- 전 세계적으로 많은 사람들이 근거리 또는 원거리 시력 손상을 가지고 있다.
- 시력 손상은 삶의 질, 이동, 사회 참여 등에 영향을 줄 수 있다.

프로젝트 연결:

- 주변 상황을 음성으로 알려주는 서비스는 시각 정보 접근이 어려운 사용자의 이동과 독립성 문제와 연결된다.

PPT 문장:

> WHO는 시력 손상이 이동, 삶의 질, 사회 참여 등에 영향을 줄 수 있다고 설명한다. 따라서 시각 정보를 음성 정보로 바꾸는 보조 서비스는 실제 필요성이 있다.

출처:

- WHO, Blindness and vision impairment  
  https://www.who.int/news-room/fact-sheets/detail/blindness-and-visual-impairment

---

### 5-2. 보건복지부 국내 등록장애인 통계

핵심 내용:

- 2024년 말 기준 국내 등록장애인은 263만 명 이상이다.
- 장애 유형 중 시각장애도 주요 비중을 차지한다.

프로젝트 연결:

- 국내에도 시각장애 관련 접근성 서비스의 대상 사용자가 존재한다.

PPT 문장:

> 보건복지부 등록장애인 통계는 국내에도 시각장애 사용자가 의미 있는 규모로 존재함을 보여준다. 이는 한국어 기반 음성 안내 서비스의 필요성과 연결된다.

출처:

- 보건복지부, 2024년 등록장애인 현황  
  https://www.mohw.go.kr/board.es?act=view&bid=0027&list_no=1485363&mid=a10503010300&nPage=1&tag=

### 5-3. 국내 시각장애인 실내보행 연구

핵심 내용:

- 시각장애인은 복잡한 실내 공간에서 독립 보행에 어려움을 느낄 수 있다.
- 공항, 지하철역, 쇼핑몰, 대형마트 등 복잡한 공간에서 독립 이동 지원의 필요성이 있다.

프로젝트 연결:

- 보이스가이드의 장애물 안내와 방향 안내 기능의 필요성을 뒷받침한다.

PPT 문장:

> 국내 실내보행 관련 연구는 시각장애인이 복잡한 실내 공간에서 독립 이동에 어려움을 겪을 수 있음을 보여준다. 보이스가이드는 이 문제를 음성 기반 주변 인지로 해결하고자 한다.

출처:

- 조성재, 김호연, 김민, 시각장애인 실내보행용 앱 개발을 위한 욕구 및 선호 연구, 시각장애연구, 2020  
  https://www.kci.go.kr/kciportal/landing/article.kci?arti_id=ART002668576

### 5-5. STT/TTS 기술 근거

핵심 내용:

- Android 등 주요 플랫폼은 음성 인식과 음성 합성 기능을 제공한다.
- STT와 TTS는 화면 조작을 줄이고 음성 중심 인터페이스를 만들 수 있게 한다.

프로젝트 연결:

- 시각장애인/저시력 사용자가 화면을 보지 않고 명령하고 안내받는 흐름에 적합하다.

PPT 문장:

> STT는 사용자의 음성을 명령으로 바꾸고, TTS는 카메라 인식 결과를 음성 안내로 전달한다. 따라서 두 기술은 화면 중심 조작을 줄이는 접근성 서비스에 적합하다.

출처:

- Android SpeechRecognizer  
  https://developer.android.com/reference/android/speech/SpeechRecognizer.html

- Android TextToSpeech  
  https://developer.android.com/reference/android/speech/tts/TextToSpeech

## 6. 기존 서비스 비교

기존 서비스 비교는 단순히 “이미 있다”로 끝내면 안 된다.

반드시 다음 흐름으로 말해야 한다.

```text
기존 서비스가 있다
  ↓
이 문제 영역은 실제 수요가 있다
  ↓
하지만 우리 서비스는 다른 초점을 가진다
  ↓
음성 명령 + 방향 안내 + 상황별 TTS가 차별점이다
```

### 비교표

| 서비스 | 주요 기능 | 강점 | 한계 또는 보이스가이드 차별점 |
|---|---|---|---|
| Google Lookout | 주변 객체/텍스트 인식 | Android 접근성 앱, 주변 설명 | 보이스가이드는 사용자의 음성 명령에 따라 장애물/물건/확인 모드를 전환 |
| Microsoft Seeing AI | 텍스트, 사람, 객체, 사진 설명 | 대표적인 AI 접근성 앱 | 보이스가이드는 실내 이동 중 방향 안내와 짧은 위험 안내를 강조 |
| Be My Eyes | 사람 또는 AI 기반 시각 도움 | 실시간 도움 연결 | 보이스가이드는 사람 연결 없이 즉시 주변 상황을 안내하는 프로토타입 |
| TapTapSee | 촬영한 물체 식별 | “이게 무엇인지” 확인에 강점 | 보이스가이드는 물체 확인뿐 아니라 찾기와 방향 안내까지 연결 |
| Envision AI | OCR, 주변 설명, 객체/사람 인식 | 다양한 시각 정보 설명 | 보이스가이드는 10일 프로젝트에 맞게 핵심 시나리오 3개에 집중 |

### 차별점 3개

1. **음성 명령 기반**
   - 사용자가 “컵 찾아줘”, “이거 뭐야?”처럼 말하면 모드가 바뀐다.

2. **방향 중심 안내**
   - 단순히 “컵”이 아니라 “컵은 왼쪽 앞에 있습니다”처럼 행동 가능한 정보를 제공한다.

3. **상황별 안내 문장**
   - 이동 중에는 짧게, 위험 상황에서는 강하게, 설명이 필요할 때는 자세히 안내한다.

## 7. 기술 타당성 검증

기술 타당성 검증은 “우리가 이 기술을 왜 쓰는가?”를 설명하는 부분이다.

### 기술 역할표

| 기술 | 역할 | 사용자 문제와 연결 |
|---|---|---|
| STT | 사용자의 음성을 텍스트 명령으로 변환 | 화면을 보며 조작할 필요를 줄임 |
| 의도 분석 | 명령어에서 원하는 기능을 판단 | “찾아줘”, “뭐야” 같은 표현으로 모드 선택 |
| 객체 탐지 | 카메라 속 물체를 인식 | 주변 장애물과 물건 파악 |
| 방향 판단 | 물체 위치를 왼쪽/정면/오른쪽으로 변환 | 사용자가 실제로 행동할 수 있는 정보 제공 |
| TTS | 안내 문장을 음성으로 출력 | 시각 정보를 청각 정보로 전달 |
| 진동 | 위험을 촉각으로 전달 | 음성만으로 부족한 상황 보완 |

### 10일 구현용 단순화

모든 기술을 완벽하게 구현하지 않아도 된다.

| 이상적인 구조 | 10일 구현용 구조 |
|---|---|
| 실제 STT 명령 입력 | STT 또는 텍스트/버튼 입력 병행 |
| 정확한 거리 측정 | 가까움/멀리 또는 방향 중심 안내 |
| 다양한 물체 인식 | 데모용 물체 3~5개만 사용 |
| 완전한 개인화 TTS | 짧은 안내/자세한 안내 템플릿 사용 |
| 실제 사용자 검증 | 팀 내부 시야 제한 기능 검증 |

발표용 문장:

> 이상적인 서비스 구조는 STT부터 TTS까지 모두 자동화하는 것이지만, 10일 프로토타입에서는 안정적인 시연을 위해 명령 입력과 거리 추정을 단순화했습니다.

## 8. 검증 전략

설문조사를 하지 않아도 검증은 가능하다.

다만 검증의 종류를 명확히 구분해야 한다.

### 검증 1. 문제 검증

방법:

- WHO, 보건복지부, 국내 연구 자료를 활용한다.

검증 내용:

- 시각장애인/저시력 사용자의 주변 인지와 이동 문제가 실제로 존재하는가?

결론:

- 문헌 자료를 통해 문제의 필요성을 확인했다.

### 검증 2. 차별성 검증

방법:

- 기존 서비스 5개를 비교한다.

검증 내용:

- 이미 비슷한 서비스가 있는가?
- 있다면 우리 서비스는 무엇이 다른가?

결론:

- 기존 서비스는 시각 정보 설명에 강점이 있고, 보이스가이드는 음성 명령 기반 방향 안내 흐름을 강조한다.

### 검증 3. 기술 적합성 검증

방법:

- STT/TTS/객체탐지/방향판단의 역할을 파이프라인으로 정리한다.

검증 내용:

- 선택한 기술이 사용자 문제 해결에 연결되는가?

결론:

- STT는 입력 접근성을 높이고, TTS는 시각 정보를 음성화하며, 객체탐지는 주변 물체 인식에 적합하다.

### 검증 4. 프로토타입 기능 검증

방법:

- 팀 내부 시야 제한 테스트를 진행한다.

검증 내용:

- 대표 시나리오 3개가 실제로 작동하는가?
- 안내 문장이 이해 가능한가?
- 실패 상황은 무엇인가?

결론:

- 대표 시나리오 작동 여부와 실패 원인을 표로 정리한다.

## 9. 팀 내부 시야 제한 테스트

이 테스트는 실제 시각장애인 사용자 검증이 아니다.

발표에서는 반드시 이렇게 표현한다.

> 실제 사용자 검증은 향후 과제로 남기고, 이번 프로젝트에서는 시야 제한 상황을 활용해 기능 흐름을 검증했습니다.

### 테스트 환경

- 온라인 팀이므로 각자 가능한 공간에서 테스트한다.
- 가능하면 한 명이 카메라/화면 공유를 하고, 다른 팀원이 결과를 기록한다.
- 테스트 물체는 인식이 잘 되는 것으로 제한한다.

추천 물체:

- 컵
- 의자
- 가방
- 책
- 텀블러

### 안전 주의

- 실제 이동 테스트는 위험할 수 있으므로 좁은 공간에서 무리하게 걷지 않는다.
- 눈을 완전히 가리고 걷는 테스트는 하지 않는다.
- 시야 제한은 화면을 보지 않고 안내를 듣는 정도로만 진행한다.

## 10. 테스트 시나리오

### 테스트 1. 장애물 안내

목적:

- 앞에 있는 물체를 음성으로 안내할 수 있는지 확인한다.

입력:

- “앞에 뭐 있어?”

기대 결과:

- “정면 가까이에 의자가 있습니다.”

측정 항목:

- 물체 인식 성공 여부
- 방향 안내 성공 여부
- TTS 출력 여부
- 안내 문장 이해 가능 여부

성공 기준:

- 5회 중 4회 이상 방향과 물체가 맞으면 성공

---

### 테스트 2. 물건 찾기

목적:

- 사용자가 찾는 물건의 방향을 알려줄 수 있는지 확인한다.

입력:

- “컵 찾아줘”

기대 결과:

- “컵은 왼쪽 앞에 있습니다.”

측정 항목:

- 목표 물체 탐지 여부
- 방향 판단 여부
- 안내 문장 출력 여부

성공 기준:

- 5회 중 4회 이상 목표 물체를 찾아 방향을 안내하면 성공

---

### 테스트 3. 물건 확인

목적:

- 카메라 중앙 또는 손에 든 물체를 설명할 수 있는지 확인한다.

입력:

- “이거 뭐야?”

기대 결과:

- “카메라 중앙의 물건은 텀블러입니다.”

측정 항목:

- 중앙 물체 선택 여부
- 물체명 안내 여부
- TTS 출력 여부

성공 기준:

- 테스트 물체 5개 중 4개 이상을 올바르게 안내하면 성공

---

### 테스트 4. 안내 방식 변경

목적:

- 짧은 안내와 자세한 안내를 구분할 수 있는지 확인한다.

입력:

- “짧게 말해줘”
- “자세히 말해줘”

기대 결과:

- 짧은 안내: “정면 장애물.”
- 자세한 안내: “정면 가까이에 의자가 있습니다. 오른쪽으로 이동하세요.”

측정 항목:

- 모드 전환 여부
- 안내 문장 길이 변화 여부
- 위험 상황에서 짧은 안내가 가능한지

성공 기준:

- 짧은 안내와 자세한 안내가 구분되어 출력되면 성공

## 12. 실패 사례와 개선 방향

실패 사례는 숨기지 않는다.

오히려 발표에서 개선 방향으로 연결하면 완성도가 높아 보인다.

| 실패 상황 | 원인 | 개선 방향 |
|---|---|---|
| 어두운 곳에서 물체 인식 실패 | 조명 부족 | 밝기 보정, 더 다양한 데이터 학습 |
| 작은 물체 인식 실패 | 화면에서 객체가 너무 작음 | 가까운 거리에서 사용하도록 안내 |
| 배경이 복잡하면 탐지 오류 | 주변 물체가 많음 | 주요 물체 우선순위 설정 |
| 안내 문장이 길어 반응 지연 | 문장 길이 과다 | 위험 상황은 짧은 안내 사용 |
| 거리 안내가 부정확함 | 단일 카메라만 사용 | 향후 깊이 추정 또는 LiDAR 연동 |

발표용 문장:

> 실패 사례를 통해 조명, 물체 크기, 배경 복잡도, 거리 추정 정확도가 한계로 확인되었고, 이는 향후 개선 방향으로 정리했습니다.

## 16. 1등을 위한 발표 전략

자료조사와 검증 파트는 다음 순서로 말한다.

```text
문제가 실제로 있다
  ↓
기존 서비스도 있다
  ↓
하지만 우리 서비스는 다른 초점이 있다
  ↓
STT/TTS와 객체 탐지가 그 문제에 맞다
  ↓
대표 시나리오로 기능을 검증했다
  ↓
실패 사례까지 분석해 개선 방향을 제시했다
```

이 흐름이 보이면 심사자가 “기술만 만든 팀”이 아니라 “문제 해결 과정을 이해한 팀”으로 볼 가능성이 높다.

> 보이스가이드는 문헌 자료와 기존 서비스 분석을 통해 문제의 필요성을 확인하고, STT/TTS와 카메라 인식을 결합한 프로토타입 테스트를 통해 대표 시나리오의 작동 흐름을 검증한 접근성 서비스이다.
