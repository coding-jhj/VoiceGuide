# 경고 피로 방지 처리 기법 가이드

## 1. 경고 피로(Alert Fatigue)란?

시각장애인 보조 앱에서 같은 경고가 너무 자주 반복되면 사용자가 경고 자체를 무시하게 되는 현상.
VoiceGuide는 이를 방지하기 위해 **5계층 다중 필터** 구조를 적용하고 있습니다.

---

## 2. 전체 계층 구조

```
YOLO 탐지
    │
    ├─[계층1] 신뢰도 < 50% 제거              (YoloDetector.kt)
    ├─[계층1] NMS로 중복 박스 통합            (YoloDetector.kt)
    │
    ├─[계층2] 보팅: 3프레임 중 2번 미만 제거  (MainActivity.kt)
    ├─[계층2] 쿨다운: 같은 사물 5초 내 차단   (MainActivity.kt)
    │
    ├─[계층3] 보팅: 10프레임 중 6번 미만 제거 (tracker.py)
    ├─[계층3] EMA: 거리 흔들림 제거           (tracker.py)
    ├─[계층3] 접근 감지: 1번만 경고            (tracker.py)
    │
    ├─[계층4] 문장 중복: 5초 내 동일 문장 → silent  (routes.py)
    ├─[계층4] alert_mode: critical/beep/silent 분류  (sentence.py)

최종 TTS 발화
```

## 3. 계층별 상세 설명

### 계층 1 — YOLO 탐지 단계

**파일**: [YoloDetector.kt](../android/app/src/main/java/com/voiceguide/YoloDetector.kt)

#### ① 신뢰도 임계값 필터 (Confidence Threshold)

```
YOLO 탐지 결과
    ├─ 신뢰도 ≥ 50% → 통과
    └─ 신뢰도 < 50% → 즉시 제거 (약한 탐지 = 오탐 가능성 높음)
```

- 적용 위치: `YoloDetector.kt:44` (`confThreshold = 0.50f`)

#### ② NMS (Non-Maximum Suppression)

```
같은 물체를 여러 박스로 탐지했을 때:
    박스A (의자, 70%)  ──┐
    박스B (의자, 55%)  ──┤ IoU > 45% → 같은 물체로 판단
    박스C (의자, 40%)  ──┘
         ↓
    박스A (의자, 70%) 만 남김 → TTS 1번만 발화
```

- 적용 위치: `YoloDetector.kt:195-208` (`iouThreshold = 0.45f`)

---

### 계층 2 — Android 앱 필터 단계

**파일**: [MainActivity.kt](../android/app/src/main/java/com/voiceguide/MainActivity.kt)

#### ③ 온디바이스 보팅 (Voting Buffer)

```
최근 3프레임 기억, 2번 이상 탐지 시 확정:

    프레임1: 의자 탐지 ✓ → 1/3 (33%) → 보류
    프레임2: 탐지 없음  → 1/3 (33%) → 보류 (오탐 가능성)
    프레임3: 의자 탐지 ✓ → 2/3 (67%) → 확정 → TTS 발화

단, 차량·칼·가위 등 위험 사물은 1프레임만 탐지해도 즉시 통과 (ALWAYS_PASS)
```

- 적용 위치: `MainActivity.kt:90-112`
- `VOTE_WINDOW = 3`, `VOTE_MIN_COUNT = 2`

#### ④ 클래스별 쿨다운 (5초)

```
"의자" TTS 발화
    └─ 5초 동안 "의자" 재탐지해도 TTS 생략
    └─ 5초 후 재탐지 시 다시 발화
```

- 적용 위치: `MainActivity.kt:99-100` (`CLASS_COOLDOWN_MS = 5000L`)

#### ⑤ 중복 박스 제거 (IoU 기반)

```
같은 클래스에서 IoU > 30% → 중복으로 판단
confidence 높은 박스만 유지
```

- 적용 위치: `MainActivity.kt:143-152`

---

### 계층 3 — 서버 추적 단계

**파일**: [tracker.py](../src/api/tracker.py)

#### ⑥ 서버 보팅 (10프레임 중 60%)

```
최근 10프레임 중 6번 이상 탐지 = 실제 물체로 확정
최근 10프레임 중 5번 이하 탐지 = 일시적 오탐으로 판단, 무시

단, 차량·위험 사물은 보팅 통과 (안전 우선)
```

- 적용 위치: `tracker.py:39-76` (`_VOTE_WINDOW = 10`, `_VOTE_THRESHOLD = 0.6`)

#### ⑦ EMA 거리 평활화

```
원본 거리:    1.0 → 1.5 → 0.8 → 1.2  (프레임마다 흔들림)
EMA 적용 후:  1.0 → 1.28 → 1.03 → 1.11 (안정적)

공식: smooth = 0.55 × 현재값 + 0.45 × 이전값
```

- 적용 위치: `tracker.py:132` (`_EMA_ALPHA = 0.55`)

#### ⑧ 접근 감지 (한 번만 경고)

```
예시: 3.0m → 2.5m → 2.0m 로 가까워짐
    └─ delta ≥ 0.4m, 거리 < 2.5m → "가까워지고 있어요" 1번만 발화
    └─ alerted 플래그 = true → 이후 더 접근해도 재발화 X

급격한 접근 (delta ≥ 0.8m, 3.0m 이내):
    → "조심! 빠르게 다가오고 있어요!" 별도 경고
```

- 적용 위치: `tracker.py:138-152`

---

### 계층 4 — 서버 문장 생성 단계

**파일**: [routes.py](../src/api/routes.py), [sentence.py](../src/nlg/sentence.py)

#### ⑨ 문장 중복 억제 (5초 캐시)

```
"의자가 있어요" 전송
    └─ 5초 이내 동일 문장 재요청 → alert_mode = "silent" 로 강제 변환
    └─ critical 경고는 5초 규칙 무시하고 항상 통과
```

- 적용 위치: `routes.py:35-52` (`_DEDUP_SECS = 5.0`)

#### ⑩ alert_mode 3단계 분류

```
critical → 즉시 TTS 음성 (1.25× 속도)
    - 계단·낙차 감지 시
    - 차량 8m 이내
    - 일반 물체 2.5m 이내
    - 동물 3m 이내

beep    → 비프음만
    - 물체 존재하나 안전한 거리

silent  → 무시
    - 멀리 있는 물체
    - 중복 억제된 경우
    - 질문 응답 후 3초 이내 periodic 경고
```

- 적용 위치: `sentence.py:36-56`

#### ⑪ 거리별 문장 긴박도 차등

```
0.5m 미만:  "위험! 정면 의자!"                      (극단으로 짧음)
0.5~1.0m:  "정면 바로 코앞에 의자가 있어요. 기다리세요."
1.0~2.5m:  "정면 약 1.5미터 앞에 의자가 있어요. 기다리세요."
2.5m 이상:  "정면 약 3미터 앞에 의자가 있어요."       (정보성)
```

- 적용 위치: `sentence.py:122-182`

## 4. 보팅(Voting) 기법 상세

### 개념

**여러 프레임에 걸쳐 같은 물체가 반복 탐지될 때만 실제 물체로 확정하는 방식.**
단 1프레임에서 탐지된 것은 오탐(빛 반사, 노이즈 등)일 수 있으므로
여러 프레임 결과를 취합(투표)하여 판단합니다.

### 원리

```
보팅 윈도우 = 5프레임, 임계값 = 60% (3번 이상)

프레임1: 의자 탐지 ✓ → 1/5 (20%) → 보류
프레임2: 탐지 없음  → 1/5 (20%) → 보류
프레임3: 의자 탐지 ✓ → 2/5 (40%) → 보류
프레임4: 의자 탐지 ✓ → 3/5 (60%) → 확정 → TTS 발화
프레임5: 의자 탐지 ✓ → 4/5 (80%) → 이미 확정됨
```

### VoiceGuide 적용 현황

| 위치 | 윈도우 | 임계값 | 파일 |
|------|--------|--------|------|
| Android 온디바이스 | 3프레임 | 2/3 (67%) | `MainActivity.kt:90` |
| 서버 tracker | 10프레임 | 6/10 (60%) | `tracker.py:39` |

### 보팅 적용 전후 성능 비교

| 항목 | 보팅 없음 | 보팅 적용 (현재) |
|------|-----------|-----------------|
| 오탐(False Positive) | 높음 (1프레임도 발화) | 낮음 (연속 탐지만 발화) |
| TTS 발화 빈도 | 초당 최대 3~5회 | 물체 확정 시 1회 |
| 응답 지연 | 없음 (즉시) | 2~3프레임 지연 (약 2초) |
| 진짜 위험 감지 | 빠름 | 차량·칼은 즉시 통과 (ALWAYS_PASS) |
| 경고 피로 | 심각 | 대폭 감소 |

### 보팅의 트레이드오프

```
장점:
  + 오탐 감소 → 경고 피로 대폭 감소
  + TTS 발화 횟수 감소 → 배터리 절감
  + 사용자 신뢰도 향상 (허위경고 없음)

단점:
  - 2~3프레임 지연 → 급박한 상황에서 다소 늦을 수 있음
  - 빠르게 지나가는 물체는 임계값 미달로 놓칠 수 있음

현재 해결책 (ALWAYS_PASS):
  차량, 오토바이, 칼, 가위, 개, 가위 등 위험 사물은
  보팅 임계값 무시하고 1프레임 탐지 시 즉시 TTS 발화
```

## 6. 설계 원칙

```
일반 물체 → 보팅 + 쿨다운 + 중복 억제로 충분히 걸러냄
위험 물체 → 모든 필터 즉시 통과 (ALWAYS_PASS + critical 모드)

목표: 경고 피로를 줄이면서도 진짜 위험은 절대 놓치지 않는다.
```
