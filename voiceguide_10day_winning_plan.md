# 보이스가이드 10일 1등용 실행 계획

## 1. 이 문서의 목적

이 문서는 처음 기획서의 풍부한 내용을 살리면서, 10일 안에 실제로 완성 가능한 **1등용 실행 계획**으로 다시 정리한 것이다.

중요한 방향:

- 기능을 많이 만드는 것이 목표가 아니다.
- 문제 정의, 사용자 분석, 고객 여정, 자료조사, 파이프라인, 검증, 데모가 하나로 연결되어야 한다.
- 10일 안에 가능한 수준으로 구현 범위를 줄이되, 발표에서는 서비스 완성도가 높아 보이게 구성한다.

최종 목표:

> 시각장애인/저시력 사용자의 주변 인지 문제를 STT-TTS 기반 서비스 흐름으로 해결하는 프로토타입을 만들고, 자료조사와 검증을 통해 설득력 있게 발표한다.

---

## 2. 프로젝트 핵심 컨셉

### 서비스명

보이스가이드

### 부제

시각장애인을 위한 음성 기반 주변 인지 보조 서비스

### 한 줄 설명

> 사용자가 말하면, 스마트폰 카메라가 주변을 인식하고, 앱이 필요한 정보를 음성 또는 진동으로 안내하는 서비스

### 핵심 시나리오

1. 장애물 안내
   - 사용자: “앞에 뭐 있어?”
   - 앱: “정면 가까이에 의자가 있습니다. 오른쪽으로 이동하세요.”

2. 물건 찾기
   - 사용자: “컵 찾아줘”
   - 앱: “컵은 왼쪽 앞에 있습니다.”

3. 물건 확인
   - 사용자: “이거 뭐야?”
   - 앱: “카메라 중앙의 물건은 텀블러입니다.”

4. 안내 방식 변경
   - 사용자: “짧게 말해줘”
   - 앱: “짧은 안내 모드로 전환합니다.”
   - 이후 앱: “정면 장애물.”

## 3. 1등을 위한 핵심 전략

7개 팀 중에서 돋보이려면 단순히 “작동하는 앱”이 아니라, **문제 해결 과정이 설득력 있는 서비스**로 보여야 한다.

### 평가 포인트 대응

| 평가 포인트 | 보이스가이드 대응 전략 |
|---|---|
| 문제 정의의 명확성 | 시각장애인/저시력 사용자의 주변 인지 문제를 한 문장으로 정의 |
| 사용자 중심 분석 | 사용자 특성, 사용 상황, 기존 방식의 한계 정리 |
| 고객 여정 및 pain point | 이동 시작부터 물건 확인까지 단계별 pain point 도출 |
| TTS-STT 활용 적합성 | 음성 명령 입력과 음성 안내 출력이 서비스 핵심 흐름 |
| 서비스 기획 완성도 | 장애물 안내, 물건 찾기, 물건 확인, 안내 방식 변경으로 구성 |
| 파이프라인 논리성 | STT → 의도 분석 → 카메라 인식 → 안내 문장 생성 → TTS |
| 시연 가능성 | 대표 시나리오 3개와 데모 영상 준비 |

### 발표에서 반드시 강조할 말

> 저희 서비스는 단순 객체 탐지 앱이 아닙니다. 사용자의 음성 의도와 카메라 인식 결과를 연결해, 시각장애인과 저시력 사용자가 필요한 정보를 음성으로 바로 받을 수 있도록 설계한 접근성 서비스입니다.

## 4. 문제 정의

### 사용자

- 시각장애인
- 저시력 사용자
- 화면 중심 스마트폰 사용이 어려운 사용자

### 사용 상황

- 낯선 실내 공간을 이동할 때
- 책상 위나 주변에서 물건을 찾을 때
- 손에 든 물건이 무엇인지 확인할 때
- 화면을 보기 어려운 상황에서 앱을 조작해야 할 때

### 주요 불편함

- 앞에 장애물이 있는지 알기 어렵다.
- 물건이 왼쪽, 오른쪽, 정면 중 어디에 있는지 알기 어렵다.
- 손에 든 물건이 맞는지 확인하기 어렵다.
- 화면을 보며 버튼을 누르는 방식이 불편하다.
- 안내가 너무 길면 이동 중 방해가 된다.

### 한 문장 문제 정의

> 시각장애인과 저시력 사용자는 기존의 화면 중심 정보 제공 방식으로 인해 주변 장애물, 물건 위치, 손에 든 물체를 즉시 파악하기 어려우므로, 이를 음성 명령과 음성 안내로 해결할 수 있는 주변 인지 보조 서비스가 필요하다.

## 5. 고객 여정 지도

| 단계 | 사용자 행동 | 사용자 생각/감정 | Pain Point | 서비스 개입 |
|---|---|---|---|---|
| 1. 이동 시작 | 낯선 실내 공간에 들어간다 | 앞에 뭐가 있는지 불안하다 | 시각 정보 부족 | 카메라 자동 인식 준비 |
| 2. 주변 확인 | “앞에 뭐 있어?”라고 말한다 | 빠르게 알고 싶다 | 화면 조작이 어렵다 | STT로 음성 명령 인식 |
| 3. 위험 탐지 | 장애물 근처로 접근한다 | 부딪힐까 걱정된다 | 위치와 거리 파악 어려움 | “정면 가까이에 의자” 안내 |
| 4. 물건 탐색 | “컵 찾아줘”라고 말한다 | 원하는 물건을 찾고 싶다 | 손으로 더듬어야 한다 | 컵 탐지 후 방향 안내 |
| 5. 물건 확인 | 물건을 카메라 중앙에 비춘다 | 이게 맞는 물건인지 궁금하다 | 비슷한 물건 구분 어려움 | “중앙 물건은 텀블러” 안내 |
| 6. 안내 조정 | “짧게 말해줘”라고 말한다 | 긴 설명이 불편하다 | 안내 과다 | 개인화 TTS 모드 변경 |

이 표는 PPT에 반드시 넣는다.

## 6. 서비스 기능 설계

### 6-1. 핵심 기능

#### 기능 1. 장애물 안내

목적:

- 사용자가 이동 중 앞에 있는 물체를 빠르게 알 수 있게 한다.

입력:

- “앞에 뭐 있어?”

출력:

- “정면 가까이에 의자가 있습니다.”
- “오른쪽에 책상이 있습니다.”
- “멈추세요. 바로 앞에 장애물이 있습니다.”

구현 범위:

- 정확한 거리는 측정하지 않는다.
- 가까움/중간/멀리 정도로만 표현한다.
- 발표 데모에서는 인식이 잘 되는 물체만 사용한다.

#### 기능 2. 물건 찾기

목적:

- 사용자가 찾고 싶은 물건의 방향을 알 수 있게 한다.

입력:

- “컵 찾아줘”
- “가방 찾아줘”

출력:

- “컵은 왼쪽 앞에 있습니다.”
- “가방은 오른쪽에 있습니다.”

구현 범위:

- 컵, 가방, 책, 의자 등 인식 가능한 소수 물체만 사용한다.
- 화면을 3등분해서 왼쪽/정면/오른쪽으로 판단한다.

#### 기능 3. 물건 확인

목적:

- 손에 든 물건이나 카메라 중앙의 물건을 확인한다.

입력:

- “이거 뭐야?”

출력:

- “카메라 중앙의 물건은 텀블러입니다.”
- “카메라 중앙의 물건은 컵입니다.”

구현 범위:

- 중앙에 가장 가까운 물체 1개만 설명한다.

#### 기능 4. 안내 방식 변경

목적:

- 사용자 상황에 따라 안내 문장 길이를 조절한다.

입력:

- “짧게 말해줘”
- “자세히 말해줘”

출력:

- 짧은 안내: “정면 장애물.”
- 자세한 안내: “정면 가까운 곳에 의자가 있습니다. 오른쪽으로 이동하세요.”
- 긴급 안내: “멈추세요. 바로 앞에 장애물이 있습니다.”

구현 범위:

- 실제 개인화 학습은 하지 않는다.
- 안내 문장 템플릿을 2~3개로 나누어 구현한다.

## 7. 전체 파이프라인

```text
사용자 음성 입력
        ↓
STT 변환
        ↓
명령 의도 분석
        ↓
모드 선택
장애물 탐지 / 물건 찾기 / 물체 설명
        ↓
카메라 이미지 분석
        ↓
객체명 + 방향 + 거리 추정
        ↓
안내 문장 생성
        ↓
개인화 TTS 출력
        ↓
음성 또는 진동 피드백
```

### 10일 구현용 단순화 버전

```text
음성 또는 텍스트 명령 입력
        ↓
명령어 키워드 분석
        ↓
카메라 객체 탐지
        ↓
왼쪽/정면/오른쪽 판단
        ↓
안내 문장 생성
        ↓
TTS 출력
```

발표에서는 전체 파이프라인을 보여주고, 구현에서는 단순화 버전을 만들었다고 설명한다.

## 8. 구현 전략

### 추천 구현 방식

온라인 5명, 10일 일정에서는 웹앱 또는 노트북 카메라 데모가 가장 현실적이다.

가능한 방식:

- 웹앱 + 카메라 + TTS
- Python OpenCV 데모 + TTS
- Android 앱 프로토타입

가장 안전한 방식:

> 웹앱 또는 Python 데모로 핵심 시나리오를 안정적으로 보여준다.

### STT 처리 전략

1순위:

- 실제 STT 구현

2순위:

- 텍스트 입력 또는 버튼으로 명령을 대신 입력

발표 설명:

> 현재 프로토타입에서는 안정적인 시연을 위해 명령 입력을 버튼/텍스트로도 지원하며, 실제 서비스에서는 STT로 대체되는 구조입니다.

이렇게 말하면 STT가 완벽하지 않아도 설계 논리가 무너지지 않는다.

### 방향 판단 전략

카메라 화면을 3등분한다.

```text
왼쪽 영역   → 왼쪽
중앙 영역   → 정면
오른쪽 영역 → 오른쪽
```

### 거리 판단 전략

객체 박스 크기로 단순 판단한다.

```text
박스가 큼   → 가까움
박스가 중간 → 중간
박스가 작음 → 멀리
```

정확한 미터 단위 거리는 하지 않는다.

## 9. 자료조사 전략

자료조사는 넓게 하지 않고, 발표에 쓸 수 있는 근거만 정리한다.

### 꼭 사용할 근거 5개

1. WHO 시각장애 및 시력 손상 자료
   - 시각 손상이 이동과 삶의 질에 영향을 줄 수 있다는 근거

2. 보건복지부 국내 등록장애인 통계
   - 국내 등록장애인 중 시각장애 비중이 존재한다는 근거

3. 국내 시각장애인 실내보행 연구
   - 복잡한 실내 공간에서 독립 보행이 어렵다는 근거

4. Google Lookout / Microsoft Seeing AI
   - 카메라 기반 시각 정보 설명 서비스가 실제로 존재한다는 근거

5. Be My Eyes / TapTapSee / Envision AI
   - 시각 정보 설명, 물건 확인, AI 도움 서비스에 대한 수요 근거

### 기존 서비스 비교표

| 서비스 | 주요 기능 | 강점 | 보이스가이드 차별점 |
|---|---|---|---|
| Google Lookout | 주변 객체/텍스트 인식 | Android 접근성 앱 | 음성 명령 기반 모드 전환 강조 |
| Microsoft Seeing AI | 텍스트, 사람, 객체 설명 | 대표 AI 접근성 앱 | 이동 상황의 방향 안내 강조 |
| Be My Eyes | 사람/AI 기반 시각 도움 | 실시간 도움 연결 | 사람 연결 없이 즉시 안내 |
| TapTapSee | 사진 속 물체 식별 | 물체 확인에 강점 | 물체 확인 + 방향 안내 |
| Envision AI | OCR, 주변 설명 | 다양한 시각 정보 설명 | 핵심 시나리오 3개에 집중 |

## 10. 검증 전략

실제 설문조사는 하지 않는다.

대신 다음 4단계로 검증한다.

1. 문헌 기반 문제 검증
2. 기존 서비스 비교 기반 차별성 검증
3. 기술 구조 기반 구현 가능성 검증
4. 팀 내부 시야 제한 테스트 기반 기능 검증

### 발표용 검증 문장

> 시간 제약상 실제 사용자 모집 대신, 문헌 조사와 기존 서비스 비교를 통해 문제의 필요성을 확인하고, 팀 내부 시야 제한 테스트로 프로토타입의 기능 흐름을 검증했습니다.

### 팀 내부 테스트

| 테스트 | 입력 | 기대 결과 | 확인할 점 |
|---|---|---|---|
| 장애물 안내 | 앞에 뭐 있어? | 장애물 방향 안내 | 방향이 맞는가, 안내가 짧은가 |
| 물건 찾기 | 컵 찾아줘 | 컵 방향 안내 | 찾는 물건을 구분하는가 |
| 물건 확인 | 이거 뭐야? | 물체 이름 안내 | 중앙 물체를 설명하는가 |
| 안내 방식 | 짧게 말해줘 | 짧은 안내 출력 | 안내 길이가 바뀌는가 |

### 실패 사례도 발표에 넣기

실패를 숨기지 않는다.

오히려 개선 방향으로 말하면 점수가 좋아질 수 있다.

| 실패 상황 | 원인 | 개선 방향 |
|---|---|---|
| 어두운 환경에서 탐지 실패 | 조명 부족 | 밝기 보정, 더 많은 데이터 |
| 작은 물체 탐지 실패 | 객체 크기 부족 | 인식 가능한 물체 범위 안내 |
| 안내가 너무 김 | 문장 길이 문제 | 긴급 상황은 짧은 문장 사용 |

## 11. 5명 회전식 역할 분담

한 명이 한 분야만 맡지 않는다.

모두가 기획, 조사, 구현, 검증, 발표를 조금씩 경험하도록 2일 단위로 역할을 회전한다.

| 기간 | A | B | C | D | E |
|---|---|---|---|---|---|
| 1~2일차 | 기획 | 자료조사 | 구현 탐색 | 검증 설계 | 발표 구조 |
| 3~4일차 | 구현 보조 | 기획 보조 | 객체 탐지 | 자료 정리 | UI/발표 |

## 13. PPT에 들어갈 핵심 문장 모음

### 문제 배경

> 시각 정보에 접근하기 어려운 사용자는 낯선 공간 이동, 물건 탐색, 물체 확인 상황에서 불편을 겪을 수 있습니다.

### 문제 정의

> 시각장애인과 저시력 사용자는 기존의 화면 중심 정보 제공 방식으로 인해 주변 장애물, 물건 위치, 손에 든 물체를 즉시 파악하기 어려우므로, 이를 음성 명령과 음성 안내로 해결할 수 있는 주변 인지 보조 서비스가 필요합니다.

### 서비스 소개

> 보이스가이드는 사용자의 음성 명령을 인식하고, 카메라로 주변 물체를 탐지한 뒤, 방향과 상황을 음성으로 안내하는 서비스입니다.

### 차별점

> 기존 서비스가 주로 시각 정보를 설명하는 데 초점을 둔다면, 보이스가이드는 사용자의 음성 의도에 따라 장애물 회피, 물건 찾기, 물건 확인을 하나의 흐름으로 연결합니다.

### 검증

> 시간 제약상 실제 사용자 모집 대신, 문헌 조사와 기존 서비스 비교를 통해 문제의 필요성을 확인하고, 팀 내부 시야 제한 테스트로 프로토타입의 기능 흐름을 검증했습니다.

### 한계와 개선

> 현재 프로토타입은 조명, 물체 크기, 카메라 각도에 따라 탐지 성능이 달라지는 한계가 있으며, 향후 실제 사용자 테스트와 거리 추정 개선이 필요합니다.

## 14. 1등을 위한 마무리 전략

7개 팀 중 돋보이려면 마지막 발표에서 다음 순서가 분명해야 한다.

```text
사용자 문제
  ↓
고객 여정과 pain point
  ↓
서비스 기능
  ↓
STT/TTS 파이프라인
  ↓
구현 데모
  ↓
검증 결과
  ↓
개선 방향
```

기술이 조금 부족해도 이 흐름이 분명하면 높은 점수를 받을 수 있다.

반대로 기능이 많아도 문제 정의와 검증이 약하면 1등은 어렵다.
